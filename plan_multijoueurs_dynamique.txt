0. CONTEXTE
------------
Le plan precedent (plan_2_machines.txt) prevoyait deux machines figees
Rouge vs Bleue, avec un scenario identique copie sur les deux postes et
une IP codee en dur au lancement. Ce modele ne repond pas a l'exigence
du sujet :

  "Vous permettez de creer une partie entre differents participants.
   Cela peut passer par un mecanisme d'invitation comme le principe
   d'une session Zoom. Vous evitez d'avoir a attendre que tous les
   participants soient presents pour commencer une partie."

Trois assouplissements decides :

  1) Scenario par joueur : chaque joueur apporte son propre fichier
     de placement (liste de SES unites). Carte partagee (memes
     dimensions), placement libre et independant.

  2) N joueurs / N equipes : plus de dichotomie R/B codee en dur.
     Chaque joueur choisit sa lettre d'equipe (A..Z). Le moteur de
     combat est deja compatible (unit.py:183 teste seulement
     self.team != other.team).

  3) Decouverte par IP + propagation de lobby : on conserve le
     modele "connaitre au moins une IP d'un pair pour rejoindre"
     (plan_2_machines.txt), fonctionne sur LAN ET sur Internet,
     pas de probleme de pare-feu broadcast. Pour permettre > 2
     joueurs sans obliger chaque nouveau a connaitre TOUS les autres,
     on ajoute la PROPAGATION DU LOBBY : quand un pair recoit un
     HELLO, il repond avec la liste des autres pairs qu'il connait
     deja. Le nouveau venu apprend alors l'existence de tout le
     groupe via UN SEUL contact initial.

     Exemple a 3 joueurs :
       - A lance en solo, joue seul
       - B lance en donnant l'IP de A -> B connait A, A connait B
       - C lance en donnant l'IP de A -> C envoie HELLO a A, A repond
         HELLO_ACK + LOBBY_LIST contenant B. C ajoute A et B a son
         propre lobby et envoie HELLO a B. Fin du handshake : chaque
         machine connait les 2 autres.

Resultat : une machine lance son jeu immediatement, joue seule en
attendant, et des qu'un autre joueur se connecte en donnant son IP
(ou l'IP de n'importe quel joueur deja dans la partie), les unites
de tout le monde convergent sur les ecrans de tout le monde (join
"a la Zoom").

4. PHASE 3 - RELAIS C : IP MANUELLE + PROPAGATION DE LOBBY
-----------------------------------------------------------
  Principe general : on conserve le mecanisme existant de main.c
  (argv[1] = IP d'un pair de reference) et on l'etend pour :
    - supporter PLUSIEURS IPs au lancement (argv[1..N])
    - envoyer un HELLO_ACK contenant la LOBBY_LIST a chaque HELLO recu
    - notifier Python quand le lobby grossit (pour resync)

  partie_c/main.c :

  4.1 Lire les IPs en argv : au lieu d'un seul argv[1], boucler sur
      argv[1..argc-1] et envoyer un HELLO unicast a chaque IP donnee.
      Les IPs sans port explicite utilisent PORT_RESEAU par defaut.
      Exemple : ./main.exe 192.168.1.20 192.168.1.21

      (Optionnel : accepter aussi une syntaxe "IP:port" pour permettre
      le multi-instance sur une meme machine avec ports differents.)

  4.2 A la reception d'un HELLO :
        - ajouter l'emetteur au lobby (add_peer)
        - construire la LOBBY_LIST = concatenation des autres IPs
          deja connues ("ip1;ip2;...") dans target_id
        - envoyer un HELLO_ACK unicast vers l'emetteur

  4.3 A la reception d'un HELLO_ACK :
        - ajouter l'emetteur au lobby
        - PARSER la LOBBY_LIST dans target_id
        - pour chaque IP nouvelle qu'on ne connait pas encore :
            - l'ajouter au lobby
            - lui envoyer notre propre HELLO (pour qu'elle nous
              ajoute a son lobby a elle)
        - cela garantit que tous les pairs se rencontrent en
          quelques round-trips, sans serveur central

  4.4 Eviter les boucles infinies : un pair ne reenvoie JAMAIS un
      HELLO_ACK a quelqu'un qu'il a deja dans son lobby avant le
      message courant. Concretement : si add_peer detecte "deja
      present" dans l'ancien etat, on ignore juste le HELLO recu
      (sauf pour le premier HELLO qui etablit la connexion).

  4.5 Notifier Python a l'ajout d'un pair : dans add_peer(), apres
      insertion effective d'une NOUVELLE adresse, envoyer un Message
      ACTION_NEW_PEER vers addr_python_out (port 5001). Le target_id
      contient l'IP du nouveau pair (informatif).

  4.6 Ports configurables via argv (optionnel mais tres utile pour
      les tests locaux a 2-3 instances sur la meme machine) :
        ./main.exe --port-ia 5000 --port-reseau 6000 192.168.1.20
      Les flags sont parses avant la liste d'IPs.

  partie_c/client.py :
    Accepter port_ecoute et port_c dans le constructeur de IPCClient
    (actuellement codes en dur 5001 / 5000).

7. PHASE 6 - NET_SYNC : MULTI-EQUIPES + RESYNC DYNAMIQUE
---------------------------------------------------------
  battle/net_sync.py :

  7.1 Encodage equipe :
      Aujourd'hui TEAM_TO_INT = {'R': 0, 'B': 1} (net_sync.py:19).
      Remplacer par :
      - id_joueur du Message = 0 (champ inutilise en multi)
      - La lettre d'equipe est le premier caractere de unit_id
        (ex "A_K_3"). _apply_spawn et autres lecteurs la recuperent
        par uid.split('_')[0].

  7.2 Filtrage a l'envoi :
      Dans send_updates(), ne traiter que les unites LOCALES :
        if unit.team != self.local_team:
            continue

      IMPORTANT : ce choix DIFFERE du plan_2_machines.txt qui voulait
      envoyer les ATTACK pour TOUTES les unites. Dans un modele N
      joueurs, c'est le PROPRIETAIRE de l'unite blessee qui est
      autoritatif sur ses HP. Quand A attaque une unite B :
        1. A calcule le degat localement, met a jour l'unite B
           (copie locale chez A)
        2. B recoit l'ATTACK de A via son propre net_sync (il voit
           que son unite a perdu des HP)
        3. Seule B broadcaste ensuite le nouveau HP a tout le monde
      Cela garantit la coherence avec N pairs : un seul emetteur
      autoritatif par unite.

      NOTE : ce schema necessite que les degats locaux soient aussi
      propages entre pairs. La solution : quand on recoit un ATTACK
      qui modifie UNE DE NOS unites, on applique le degat ET on laisse
      le prochain send_updates() le renvoyer comme event local. Les
      doublons (2 chemins) sont absorbes car le HP est ABSOLU, pas
      differentiel.

  7.3 Resync sur nouveau pair :
      Dans _apply(), ajouter :
        elif msg.action == ActionType.NEW_PEER:
            self.prev_snapshot = {}   # force full resend au prochain tick
      Les pairs deja au courant ignorent les duplicatas via
      find_unit_by_id (net_sync.py:116).

  7.4 Ports configurables :
      NetSync instancie IPCClient() avec les ports passes par
      engine.init_network (plus de hardcoding).

-------------------------

    --team A              lettre d'equipe locale (active le mode multi)
    --map-size 100x100    dimensions de la carte partagee
    --scenario-file PATH  fichier de placement local
    --port-c N            port du relais C cote envoi (defaut 5000)

    ./partie_c/main.exe

        --scenario-file data/scenario/armee_a.txt Major_DAFT

    # Joueur B (rejoint A en connaissant son IP)
    ./partie_c/main.exe 192.168.1.10
    python main.py battle run --team B --map-size 100x100 \
        --scenario-file data/scenario/armee_b.txt Major_DAFT

    # Joueur C (rejoint la partie en connaissant JUSTE l'IP de A,
    # il decouvre B automatiquement via la LOBBY_LIST)
    ./partie_c/main.exe 192.168.1.10
    python main.py battle run --team C --map-size 100x100 \
        --scenario-file data/scenario/armee_c.txt Major_DAFT

11. PIEGES CONNUS
------------------
  - Paquets UDP perdus -> unite zombie distante. Mitigation legere :
    renvoyer DEATH quelques ticks de suite (evolution future, pas
    dans ce plan).
  - Desynchronisation des dimensions : si deux joueurs passent des
    --map-size differents, un SPAWN arrivant avec une position hors
    carte plante. Ajouter une verif dans _apply_spawn qui rejette les
    positions out-of-bounds + log.
  - Collision de lettres d'equipe : si deux joueurs choisissent
    --team A, leurs unites auront des IDs qui peuvent collisionner
    (A_K_0 chez les deux). Mitigation v1 : laisser l'utilisateur
    s'organiser. v2 : rejeter au HELLO si un pair declare deja notre
    lettre (le HELLO peut porter la lettre d'equipe dans target_id).
  - Latence LAN : 1-5ms. A 60 TPS (16ms/tick), decalage visuel de
    1-2 ticks, imperceptible.
  - LOBBY_LIST trop longue : target_id limite a 64 chars, soit environ
    4 IPv4 "192.168.1.100;" par message. Au-dela de 4 pairs connus,
    fragmenter la LOBBY_LIST sur plusieurs HELLO_ACK consecutifs.
  - Chaque pair doit etre JOIGNABLE depuis les autres (port UDP
    RESEAU ouvert en entrant). Sur Internet derriere NAT, cela
    necessite une redirection de port ou un pair de rendez-vous
    (hors scope).
